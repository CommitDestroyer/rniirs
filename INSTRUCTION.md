# Инструкция по реализации кейса РНИИРС: Мониторинг спутников

**Проект:** Satellite Monitor Platform (RNIIIRS Case)
**Цель:** Разработка веб-приложения для 3D-трекинга спутников в реальном времени с использованием TLE-данных, библиотеки satellite.js и CesiumJS.
**Стек технологий:** Ubuntu/Linux, Node.js, React, CesiumJS (для 3D-глобуса).
**Целевая аудитория:** Инженеры и операторы спутниковых систем.

## 1. Описание проекта (Project Overview)
В рамках кейса РНИИРС необходимо разработать высокопроизводительную платформу для визуализации и мониторинга околоземных объектов. Платформа должна обеспечивать прием, парсинг и обработку Two-Line Element (TLE) данных для точного расчета орбит и положения спутников в пространстве. Решение направлено на повышение эффективности контроля за орбитальной группировкой и предоставление удобного инструмента визуального анализа.

## 2. Архитектура системы (Architecture)
Система строится на базе современных веб-технологий и модульной архитектуры:
*   **Frontend (UI/UX & 3D Engine):** React для управления состоянием приложения и пользовательским интерфейсом. Интеграция CesiumJS для рендеринга 3D-глобуса и пространственных объектов с высокой производительностью.
*   **Math Core (Орбитальная механика):** Использование библиотеки `satellite.js` для глубокой математической обработки. Преобразование сырых TLE-данных в координаты ECI (Earth-Centered Inertial) и последующая конвертация в ECEF (Earth-Centered, Earth-Fixed) или геодезические координаты (Lat/Lon/Alt) для отображения на глобусе.
*   **Data Pipeline:** Асинхронное получение актуальных TLE-каталогов из открытых источников (например, Celestrak), парсинг и нормализация данных перед передачей в математическое ядро.

## 3. Дорожная карта разработки (Feature Roadmap)

### Phase 1: Базовый функционал (Globe & 100+ Satellites rendering)
*   Инициализация сцены CesiumJS (отключение лишних виджетов, настройка освещения).
*   Загрузка тестового набора TLE-данных.
*   Расчет координат и рендеринг более 100 спутников на орбите в виде point/billboard примитивов.

### Phase 2: Аналитика объектов (Orbit trajectories & Info panels)
*   Отображение орбитальных траекторий (путь спутника за один или несколько витков).
*   Интерактивные информационные панели (Info panels) при клике на объект: отображение имени, номера NORAD, текущей высоты, скорости и координат (Lat/Lon).

### Phase 3: Расчет зон радиовидимости (Visibility calculations)
*   Реализация функционала расчета времени пролета (Overpass times) объектов над заданной наземной станцией.
*   Ввод пользовательских координат (Lat/Lon) наземной станции и отображение визуальной связи (линии видимости) при попадании спутника в зону радиовидимости с учетом угла места (elevation angle).

### Phase 4 (Bonus): Продвинутая аналитика (Ground footprint visualization & Doppler shift charts)
*   Визуализация площади покрытия на поверхности Земли (Ground footprint) динамическим полигоном или эллипсом.
*   Расчет и построение графиков доплеровского смещения частоты (Doppler shift) в зависимости от взаимного положения приемника и передатчика.

## 4. Технические ограничения (Technical Constraints)
*   **FPS Stability:** Плавный рендеринг (не менее 30-60 FPS) при отображении большого количества объектов. Использование Entity API / Primitive API Cesium оптимальным образом.
*   **Offline TLE Caching:** Реализация механизма кэширования TLE-данных в LocalStorage/IndexedDB для работы платформы в условиях нестабильного интернета или оффлайн-режиме, а также для снижения нагрузки на внешние API.
*   **Responsive UI:** Адаптивный интерфейс панелей управления, корректно отображающийся на различных разрешениях экранов рабочих станций.

## 5. Правила работы с репозиторием (Repository Rules)
*   **Clean Code:** Код должен соответствовать установленным стандартам `eslint/prettier`, не содержать мертвых участков и "магических чисел".
*   **Modularity:** Жесткое разделение логики: UI компоненты отдельно, математические сервисы (satellite wrappers) отдельно, API клиенты отдельно.
*   **Documentation:** Все публичные методы, компоненты и сложные математические преобразования должны быть задокументированы в формате JSDoc.
*   **SourceCraft Submission:** Обязательная проверка работоспособности сборки, покрытие ключевых модулей модульными тестами и проверка документации перед каждым коммитом для финальной сдачи в систему контроля версий.
